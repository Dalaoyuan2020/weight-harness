# Harness Engineering 宣言

> 三个臭皮匠，顶个诸葛亮。

---

## 什么是 Harness Engineering？

Harness Engineering 的核心，不是模型的神机妙算或算无遗策，不是要求模型一把就把所有问题都解决了，给出一个绝妙方案后通过精准执行达到 99.9% 的效果。

我们的逻辑是：

1. **先跑通 10%**，保住这部分成果。
2. **再完成 30%**，通过不断的收敛性迭代，让项目快速达到 80% 的完成度。

Harness 的价值不在于它的精准或微妙，而在于它的**收敛性**。

> 智者千虑，必有一失；愚者千虑，必有一得。

我们追求的是通过多次实验，摸索出一条最正确的路径。通过多模型一边跑一边迭代，最终让它**收敛**成一个可用且具有健壮性的东西，而不是靠某种"金标式"的一次性完美设计。

## 三方制衡

Harness Engineering 讲究的是**三个层之间的相互监督**：

```
        🧠 编排层
       ╱        ╲
      ╱  相互监督  ╲
     ╱              ╲
💪 执行层 ———————— 👁️ 评估层
```

- **编排层**（教练/主脑）：制定计划，调度资源
- **执行层**（学生/Worker）：接收指令，完成任务
- **评估层**（家长/验收者）：监督进度，评估质量

三者互相制衡，没有任何一方能独断。就像三权分立——编排不能自己评估自己，执行不能自己验收自己，评估不能自己安排自己。

## 五层架构

```
┌─────────────────────────────────────┐
│  L5  操作器（Agent + 定时器）        │  触发联动
├─────────────────────────────────────┤
│  L4  连接层（开发者协议）            │  连接各端口的自动化工具
├─────────────────────────────────────┤
│  L3  评估层（家长端）                │ ╲
├─────────────────────────────────────┤  ├─ 主体层：相互监督
│  L2  编排层（教练端）                │ ╱
├─────────────────────────────────────┤╱
│  L1  执行层（学生端）                │
└─────────────────────────────────────┘
```

### 主体层（L1 + L2 + L3）
编排、评估与执行三个端，基于通信层实现相互监督。

### 连接层（L4）
作为连接各个端口的自动化工具，也是我们说的**开发者协议**。定义了三端之间怎么通信、数据怎么流转、接口怎么调用。

### 操作器（L5）
引入一个 Agent。通过 L4 将三个端连接起来，再通过 L5 的 Agent 去触发联动。怎么联动？在 L5 上加一个**定时器**即可。

## 收敛，而不是乱跑

整个自动化框架的本质并不复杂。通过这套框架让模型跑起来，讲究的是一种**「收敛的跑」**，而不是乱跑。

每一轮迭代：
1. 编排器下发计划
2. 执行者执行并反馈
3. 评估器验收并提出调整
4. 编排器根据反馈修正计划
5. 循环往复，不断收敛

```
迭代1: ████░░░░░░ 10%  — 先跑通
迭代2: ██████░░░░ 30%  — 保住成果
迭代3: ████████░░ 50%  — 持续收敛
迭代4: █████████░ 80%  — 可用且健壮
```

**这就是 Harness Engineering。**
